Project Compliance & Business Logic Specification

（項目合規與業務邏輯規範）

1. 文檔目的（Purpose）

本文件定義本項目在以下維度的強制規範與判斷邏輯：
 • 合法性（法律、消保、稅務）
 • 平台規則（App Store / 金流）
 • 業務可行性（現金流、風險、可擴展性）
 • 產品表現形式（UI / UX / 文案）

本文件同時作為：
 • LLM 理解與生成行為的約束依據
 • 項目審計與風險回報的判斷標準
 • 業務模型調整時的修正參考

⸻

2. 核心原則（Global Principles）

2.1 交易與收入原則
 1. 交易主體即責任主體，責任主體即收入歸屬主體。
 2. 非屬平台之資金，不得列為平台營業收入。
 3. 條件未成立前之收款，不構成成交。
 4. 可拆分之交易，必須拆分為不同性質之收費項目。

2.2 App 與平台規則原則
 1. App 內不得販售非允許之數位商品形式。
 2. App 內僅能販售：
 • 平台自身數位服務（訂閱 / 功能）
 • 實體服務之預約或保留（非票券）
 3. App 內不得生成可交易、可轉讓之數位資產。

2.3 風險控制原則
 1. 高風險行為必須條件化、限額化、狀態化。
 2. 風險需前置於制度與系統，而非事後人工處理。
 3. 風險不得集中於平台單一主體。

⸻

3. 收費與金流類型定義（Authoritative Definitions）

3.1 名額保留費（Conditional Reservation Fee）

定義
用於保留實體活動入場名額之條件式履約保證款。

法律性質
 • 成團條件成立前：非交易、非收入
 • 成團條件成立後：折抵實體服務價金

必要條件
 • 明確成團條件（人數 / 金額 / 期限）
 • 條件未成立自動全額退款
 • 不生成票券、不生成資產
 • 不可轉讓、不可拆分

會計處理
 • 成團前：暫收款 / 受託款
 • 成團後：依拆分規則認列服務費部分

禁止事項
 • 不可稱為票券、數位資產、投資、募資
 • 不可設定不可退條款（於成團前）

⸻

3.2 平台服務費（Platform Service Fee）

定義
平台就媒合、管理、票務、通知等服務所收取之費用。

法律性質
 • 已提供服務即構成成交
 • 屬平台營業收入

會計與稅務
 • 僅就服務費開立發票
 • 不認列活動全額 GMV

⸻

3.3 訂閱型數位服務（Subscription）

定義
平台提供之數位功能、權限或工具。

平台規則
 • App 內必須使用 In-App Purchase
 • 適用 Small Business Program 抽成（如符合）

風險控制
 • 不與實體活動金流混合
 • 不與名額保留費綁定

⸻

3.4 廣告、曝光、贊助（Marketing / Sponsorship）

定義
品牌或主辦方為取得曝光或行銷效果所支付之費用。

法律性質
 • 行銷服務
 • 非交易對價

執行原則
 • 優先 Web / 合約制
 • 不直接與入場權或名額綁定

⸻

3.5 商品與創作者收入（QuoteMe）

允許形式
 • 平台自營實體商品
 • 智慧財產授權 / 設計授權

禁止形式
 • App 內打賞
 • App 內多方分帳
 • 數位資產化創作物

⸻

4. 風險分類與應對策略（Risk Matrix）

4.1 高風險行為
 • 未履約前收取不特定多數金錢
 • 平台長期保管非自身資金
 • App 內生成可交易數位物件

應對方式
 • 條件化
 • 限額化
 • 狀態化
 • 第三方金流或信託

⸻

4.2 中風險行為
 • 延後結算
 • 免費主辦預售

應對方式
 • 主體驗證
 • 金額、人數、期間上限
 • 履約責任前置同意

⸻

5. 系統與產品表現約束（Implementation Rules）

5.1 UI / 文案限制

禁止使用詞彙
 • 票券
 • 購買
 • 擁有
 • 數位資產
 • 投資 / 募資

允許使用詞彙
 • 名額保留
 • 成團確認
 • 履約條件
 • 狀態成立

⸻

5.2 狀態機模型（必須）

所有收款相關行為必須對應狀態，而非物件：
 • pending_condition
 • condition_met
 • refunded
 • fulfilled

⸻

6. LLM 行為約束（For AI Systems）

當 LLM 檢視或生成新功能時，必須：
 1. 判斷該功能是否涉及金流
 2. 判斷是否構成成交
 3. 指出可能違反之平台或法律原則
 4. 提出至少一個低風險替代方案
 5. 標示該方案對業務的影響（轉換率 / 收入 / 風險）

若無法同時滿足合法性與業務目標，必須回報衝突。

⸻

7. 審計與修正回報格式（Standard Output）

當發現風險時，請依以下格式回報：
 • 問題描述
 • 涉及規範條款
 • 風險等級（高 / 中 / 低）
 • 可能後果
 • 修正方案（至少一個）
 • 對業務的影響評估

⸻

8. 核心結論（Final Constraint）

本項目允許「有代價的確認」，
但不允許「未成交即被視為銷售」。

本項目的設計目標是：
降低不確定性，而非集中風險；
讓資金流動，而非進帳；
讓責任可追，而非平台承擔。
